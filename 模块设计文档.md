# 学术论文智能搜索系统 MVP 模块设计文档

## 项目概述

为校赛"科研场景下复杂学术查询的智能论文搜索与推荐"赛题构建最小可用版本（MVP）。目标是端到端跑通全流程：用户输入自然语言查询 → 系统自动理解、检索、排序、输出结构化结果。先跑通再由队友优化，本人主攻检索策略研究。

技术栈：Python + glm-4.7-Flash Flash API + Semantic Scholar API + OpenAlex API

## 项目结构

```
PaperSearch/
  .env                    # GLM_API_KEY=xxx（智谱API Key）
  .gitignore
  requirements.txt        # openai, requests, python-dotenv
  main.py                 # 入口：CLI 交互，串联全流程
  api_wrapper.py          # 模块①：封装 Semantic Scholar + OpenAlex API
  query_understanding.py  # 模块②：LLM 查询理解与扩展
  relevance_assessment.py # 模块③：LLM 相关性评估
  agent_strategy.py       # 模块④：Agent 调度策略（串联①②③）
  output_formatter.py     # 模块⑤：结构化 JSON 输出
```

## 各模块设计

### 模块① api_wrapper.py — 学术搜索 API 封装

**职责：** 封装学术搜索 API 调用，统一返回格式

**主要函数：**
- `search_papers(query, limit=20) -> list[dict]`
  - 优先使用 Semantic Scholar，失败时自动降级到 OpenAlex
  - 返回统一格式：`{paperId, title, abstract, year, authors, citationCount, url}`

**技术细节：**
- Semantic Scholar：`GET https://api.semanticscholar.org/graph/v1/paper/search`
  - 支持 `x-api-key` 头部提高速率限制（免费申请）
  - 429 限流时指数退避重试（1s → 2s → 4s）
- OpenAlex（备选）：`GET https://api.openalex.org/works`
  - 无需 API Key，需设置 User-Agent
  - 摘要格式为 inverted index，需重建为文本

**优化方向（交给队友）：**
- 增加 OpenAlex 作为并行搜索源（而非仅降级备选）
- 添加更多 API（PubMed、DBLP 等）
- 搜索结果缓存机制

---

### 模块② query_understanding.py — 查询理解与扩展

**职责：** 用 LLM 解析用户查询意图，生成多组搜索词

**主要函数：**
- `expand_query(user_query) -> dict`
  - 返回：`{original_query, entities: {methods, domains, key_terms}, search_queries: [3-5组]}`

**LLM 调用参数：**
- 模型：`glm-4.7-Flash Flash`
- 温度：0.3（确定性为主，允许少量变化）
- 强制 JSON 输出

**Prompt 核心要求：**
- 生成 3-5 组搜索词，每组 2-8 个词
- 覆盖不同角度：精确术语、同义词、更广领域、特定方法
- 适合关键词搜索（非自然语言句子）

**兜底机制：** JSON 解析失败时用原始查询作为唯一搜索词

**优化方向（交给队友）：**
- 支持中英文查询
- 更细粒度的实体抽取（时间范围、venue、数据集名）
- 查询分类（是方法导向、问题导向还是综述型）

---

### 模块③ relevance_assessment.py — 相关性评估

**职责：** 用 LLM 判断每篇论文与用户查询的相关程度

**主要函数：**
- `assess_relevance(user_query, papers) -> list[dict]`
  - 为每篇论文添加：relevance、relevance_score（0-1）、relevance_reason

**LLM 调用参数：**
- 模型：`glm-4.7-Flash Flash`
- 温度：0.1（尽量确定性）
- 批量处理：每 8-10 篇一次调用

**评分标准：**
| 等级 | 分数范围 | 含义 |
|------|---------|------|
| highly_relevant | 0.7-1.0 | 直接解决查询核心问题 |
| partially_relevant | 0.3-0.7 | 相关领域或仅涉及部分方面 |
| not_relevant | 0.0-0.3 | 关系松散 |

**优化方向（交给队友）：**
- 利用全文信息（不仅是标题+摘要）
- 考虑引用关系（被相关论文引用的论文更可能相关）
- 加入时效性权重（近年论文优先）

---

### 模块④ agent_strategy.py — Agent 调度策略

**职责：** 串联模块①②③，控制搜索流程

**当前策略（MVP 最简版）：**
1. 调用模块②：生成 3-5 组搜索词
2. 调用模块①：每组搜索词搜索一次，合并去重
3. 调用模块③：批量评估相关性
4. 过滤 not_relevant → 按 score 降序 → 取 top_k

**这是需要本人重点研究的模块，后续升级方向：**
- 多轮迭代搜索（根据已有结果动态调整关键词）
- 引文网络追踪（找到A论文后顺藤摸瓜）
- 查询分解（复杂查询拆为子问题分别搜索）
- 停止条件判断（搜到什么程度算够了）

---

### 模块⑤ output_formatter.py — 结构化输出

**职责：** 将结果格式化为结构化 JSON

**输出结构：**
```json
{
  "query": {
    "original": "用户原始查询",
    "entities": {"methods": [], "domains": [], "key_terms": []},
    "expanded_queries": ["扩展搜索词1", "扩展搜索词2", ...]
  },
  "summary": {
    "total_found": 28,
    "highly_relevant": 8,
    "partially_relevant": 12
  },
  "results": {
    "highly_relevant": [{论文详情}, ...],
    "partially_relevant": [{论文详情}, ...]
  }
}
```

**优化方向（交给队友）：**
- 生成关系图/知识图谱
- 自动生成综述性总结
- 按子主题分组展示

## 数据流

```
用户输入查询（自然语言）
       ↓
  模块② query_understanding
  LLM 生成 3-5 组搜索词 + 提取实体
       ↓
  模块① api_wrapper（×3-5 次）
  每组搜索词调用学术 API
       ↓
  模块④ agent_strategy
  合并去重（约 25-35 篇）
       ↓
  模块③ relevance_assessment（批量 LLM 调用）
  为每篇论文评分排序
       ↓
  模块④ agent_strategy
  过滤 + 排序 + 取 top_k
       ↓
  模块⑤ output_formatter
  结构化 JSON 输出
       ↓
  main.py 打印摘要 + 保存文件
```

## API Key 配置

在 `.env` 文件中配置：
```
GLM_API_KEY=你的智谱APIKey          # 必填，open.bigmodel.cn 获取
SEMANTIC_SCHOLAR_API_KEY=你的Key    # 选填，提高搜索速率
```
